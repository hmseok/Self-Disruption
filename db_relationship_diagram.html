1. 통장관리 → 분류 → 차량매칭 전체 흐름
1
통장/카드 거래내역 업로드
엑셀 파일 파싱 → Gemini AI 분석 → transactions 테이블에 INSERT
POST /api/finance-parser → INSERT INTO transactions
↓
2
AI 자동 분류 (classification_queue)
거래처명 매칭 + AI 카테고리 분류 → 신뢰도 80% 이상 자동확정, 미만 수동검토
POST /api/finance/classify → INSERT INTO classification_queue
↓
3
차량/계약 매칭
분류 결과에 따라 related_type + related_id 설정 (차량, 지입, 투자, 대출 등)
UPDATE transactions SET related_type='car', related_id='{car_id}'
↓
4
정산/리포트
차량별 수익/비용 집계, 지입 수익배분, 투자이자 계산, 손익계산서
SELECT * FROM transactions WHERE related_type='car' AND related_id=?
2. transactions 테이블 (핵심 — 모든 거래의 중심)
transactions 핵심 테이블
PK id UUID
FK company_id UUID → companies(id)
transaction_date DATE 거래일 (시간 없음)
client_name TEXT 거래처명
description TEXT 적요/메모 (중복체크 키)
amount NUMERIC(15,2) 금액
type VARCHAR(20) income / expense
status VARCHAR(20) draft/pending/completed/cancelled
payment_method VARCHAR(20) bank_transfer / card / cash
── 차량매칭 핵심 컬럼 ──
category TEXT 유류비, 정비/수리비, 급여 등
related_type TEXT 다형성 FK 타입
related_id TEXT 다형성 FK ID
classification_source TEXT manual / rule / ai
confidence INT AI 분류 신뢰도 0-100
── 기타 참조 컬럼 ──
FK card_id UUID → corporate_cards(id)
FK employee_id UUID → profiles(id)
employee_name TEXT 직원명 (비정규화)
is_cancel BOOLEAN 취소 거래 여부
cancel_pair_id UUID 취소-승인 매칭
currency VARCHAR(10) KRW/USD/JPY/EUR
original_amount NUMERIC(15,2) 외화 원금액
memo TEXT 비고
3. related_type / related_id 다형성(Polymorphic) 관계 — 핵심 매칭 구조

transactions 테이블은 별도 브릿지 테이블 없이 related_type + related_id 2개 컬럼으로 여러 테이블을 동적으로 참조합니다.

'car' → cars.id
유류비, 정비/수리비, 차량보험료, 자동차세, 통행료/주차비, 차량매입
'jiip_share' → jiip_contracts.id
지입 수익배분금(출금) — 정산 시 생성
'invest' → general_investments.id
투자자 이자 지급 — 정산 시 생성 (id=BIGINT)
'loan' → loans.id
차량할부/리스료, 원금상환, 이자비용
'salary' → employee_salaries.employee_id
급여(정규직), 상여금/퇴직금
'freelancer' → freelancers.id
용역비(3.3%) — 프리랜서 용역비
'insurance' → insurance_policy_record.id
차량보험료, 보험금 수령
'card' → corporate_cards.id
법인카드 거래 — card_id FK와 별도
'contract' → contracts.id
렌탈계약 관련 수입/지출
4. classification_queue (분류 검토 큐)
classification_queue 분류/매칭 브릿지
PK id UUID
FK company_id UUID → companies(id)
FK transaction_id UUID → transactions(id) ON DELETE CASCADE
source_type TEXT 'bank_statement' (기본값)
source_data JSONB 원본 거래 데이터 스냅샷
── AI 분류 결과 ──
ai_category TEXT AI 추천 카테고리
ai_confidence INT 0-100 (80↑ 자동확정)
ai_related_type TEXT AI 추천 연관타입 (car, jiip 등)
ai_related_id TEXT AI 추천 연관ID
alternatives JSONB 대체 분류 후보 [{category, confidence}]
── 사용자 확정 결과 ──
final_category TEXT 최종 확정 카테고리
final_related_type TEXT 최종 확정 연관타입
final_related_id TEXT 최종 확정 연관ID
status TEXT pending → auto_confirmed / confirmed / rejected
FK reviewed_by UUID → profiles(id)
classification_queue ↔ transactions 관계:

classification_queue.transaction_id → transactions.id (1:1 또는 1:N)

v1 방식: transaction_id로 직접 참조 (FK ON DELETE CASCADE)
v3 방식: source_data JSONB에 원본 스냅샷 저장 (transaction 삭제되어도 이력 유지)

확정 시: classification_queue의 final_* 값을 → transactions의 category, related_type, related_id에 UPDATE

7. 전체 FK 관계 요약 (화살표 = FK 방향)
transactions ├─ company_id ───FK──→ companies ├─ card_id ───FK──→ corporate_cards ├─ employee_id ───FK──→ profiles ├─ related_type='car' + related_id ──→ cars.id ├─ related_type='jiip_share' + related_id ──→ jiip_contracts.id ├─ related_type='invest' + related_id ──→ general_investments.id (BIGINT!) ├─ related_type='loan' + related_id ──→ loans.id ├─ related_type='salary' + related_id ──→ employee_salaries.employee_id ├─ related_type='freelancer' + related_id ──→ freelancers.id ├─ related_type='insurance' + related_id ──→ insurance_policy_record.id └─ related_type='card' + related_id ──→ corporate_cards.id classification_queue ├─ transaction_id ───FK──→ transactions.id (CASCADE) └─ ai/final_related_type + _id ──→ (위 다형성 관계와 동일) cars ←── 하위 테이블들 ├── car_costs.car_id (취득비용 항목) ├── jiip_contracts.car_id (지입계약) ├── loans.car_id (대출/할부) ├── maintenance_records.car_id (정비이력) ├── accident_records.car_id (사고이력) ├── inspection_records.car_id (검사이력) ├── vehicle_operations.car_id (출고/반납) └── vehicle_schedules.car_id (스케줄) transaction_flags ├─ transaction_id ──FK──→ transactions.id ├─ queue_id ──FK──→ classification_queue.id └─ salary_adjustment_id──FK──→ salary_adjustments.id
8. 중복체크(Dedup) 키 구성 — 수정 완료
기존 (문제): transaction_date | client_name | |amount| | payment_method
→ 같은 날 같은 금액 이체시 정상 거래가 중복으로 삭제됨

수정 후: transaction_date | client_name | |amount| | payment_method | description
→ 적요(이체시간, 메모)가 다르면 별건으로 인식
