## 2026-07-08 PR-SMS-LEDGER — 재파싱=원장 채움 (연동 미등록 환경 대응)
- **진단**: 오픈뱅킹 연동 계좌 등록 0건 (codef_connections 비어있음, 동기화 기록 0) — 문자가 통장의 유일한 입구임을 확인. 정본 정책은 「연동 있을 때만」 발동이라 데이터 구멍은 없음.
- **재파싱 확장**: 실패→성공으로 살아난 **은행 문자는 중복 검사 후 통장 거래로 바로 등록** (잔액 포함 → 계좌별 잔액 사슬 자동 검증에 합류). 판별자 프리로드(같은 날+금액+입출+잔액, 없으면 +적요)로 대량에도 빠름. 결과창에 「등록 N건 / 이미 있어 제외 K건」. 은행 입금 등록 시 대차 자동매칭 1회 트리거.
- **계좌 꼬리 해석 `resolveAccountLast4`** (lib/last4-match 공용): KB 문자 계좌 마스킹(217001**168 — 꼬리 3자리)을 매핑 테이블 계좌번호와 대조해 실제 끝4자리(6168)로 승격. webhook/재파싱/개별 등록 3곳 동형 적용 (규칙 14). 잔액 검증 계좌 필터도 3자리 꼬리 행 포함.

## 2026-07-08 PR-SMS-RECOVER — 카운트 정상화 + 문자 개별 복구
- **목록 로드 한도 5,000 → 20,000**: 통장 12,000+건 환경에서 절반(옛날 쪽)만 로드되어 필터 카운트 「전체 5000」·계좌 잔액 3/19 기준으로 잘리던 사고 수정. 화면 카운트·계좌 카드·잔액 검증이 전체 데이터 기준으로 정상화.
- **KB은행 새 문자 형식 파서**: `[KB]07/08 13:50 217001**168 적요 출금 20,000 잔액26,149,599` (원 없음·계좌 마스킹·적요 포함) 형식 추가 — 파싱 실패 1,108건의 원인. 「실패 건 재파싱」 실행 시 일괄 회복. 적요(차량번호 등)를 가맹점 칸에 저장 → 매칭 재료. webhook 잔액 추출도 「원」 없는 형식 대응.
- **은행 연동(오픈뱅킹) 중복 검사 신설**: 기존엔 존재 확인 없이 INSERT — 30분 주기가 같은 3일치를 재등록할 구조적 위험. 같은 날+금액+입출+잔액(없으면 거래처) 판별자로 통장 3계열 전체와 대조 후 등록. 삭제된 건(deleted)은 대조에서 제외 → **화면에서 삭제한 오늘 거래는 다음 주기에 자동 복구**.
- **SMS 수집 개별 「거래로 등록」 버튼**: 파싱 성공인데 거래가 삭제/부재인 건에만 표시 (연결 살아있으면 「연결됨」). 등록 전 중복 검사, 등록 후 대차 자동매칭 트리거 (webhook 동형). 전체삭제로 사라진 문자 건 개별 복구용.
- **엑셀 업로드 교차 중복 차단**: 엑셀이 연동(codef)/은행문자가 이미 등록한 거래를 다시 넣지 않게 같은 판별자(날짜+금액+입출+잔액)로 대조 — 3방향(엑셀↔엑셀, 연동→엑셀, 엑셀→연동) 모두 차단 완성. 결과창 skip 사유에 cross_source 추가.
- SMS 금액 표시 `+` 부호 제거 (규칙 18 — 색상으로 의미 표현).
