## 영향

```
✅ 코드/마이그레이션/_docs 모두 main 에 반영 (긍정적)
⚠ commit 메시지가 "RideAccidents PR-6.3.c" 로 되어있어 PR-2SS 시리즈 추적 어려움
⚠ 두 PR (RideAccidents + PR-2SS) 가 한 commit 에 묶여 git history 분리 불가
⚠ 만약 RideAccidents 부분 revert 필요해지면 PR-2SS 도 같이 revert 됨
⚠ 세션 B 가 자기 작업을 "이미 push 됨" 으로 인지 시 별도 commit 만들지 않게 됨
   → PR-2SS 의 commit 메시지/설계서/CHANGELOG 정리 누락 위험
```

## Root Cause (3-Why)

```
WHY 1: 사용자가 git add . 실행 → 다른 세션 작업물도 staged
WHY 2: 본 세션 push 안내가 명시 path 였으나 자동 강제 X — 실수 인지 못함
WHY 3: cowork-staging-lint 자동화 hook 미구현 → 시스템 차원 차단 부재
```

근본 원인 = **자동화 hook 부재**. 인간 주의력 / 명령 정확성 의존.

## Prevention (재발 방지)

### 1. 자동화 hook 신설 ✅ (본 PR-Z1)
```
harness-engineering/scripts/cowork-staging-lint.js — 신설
   - staged 파일 목록 추출 (git diff --cached --name-only)
   - 각 파일을 모듈 영역에 매핑
   - 모듈 set 의 size > 1 + 화이트리스트 외 → commit 차단
   - 화이트리스트 (한 PR 에 여러 모듈 가능):
     · app/components/* — 공통 컴포넌트
     · lib/* — 공통 라이브러리 (모듈 전용 *-db.ts 등 제외)
     · harness-engineering/* — 하네스 인프라
     · prisma/*, package.json, CLAUDE.md, HARNESS.md, root 설정

기존 harness-lint.js 의 sub-lint 통합:
   const coworkStagingLint = require('./cowork-staging-lint')
   awaits coworkStagingLint() 후 결과 집계
```

### 2. 명시적 GO 우회 패턴
```
의도적인 cross-module commit 시:
  - commit 메시지 prefix: "[multi-module]" 또는 "[cross]"
  - 또는 환경변수: COWORK_ALLOW_MULTI_MODULE=1 git commit ...
```

### 3. 사용자 push 안내 강화
```
세션 별 push 안내 시 다음 형식 강제:
  ⚠ 본 세션 영역만 staging — git add . 절대 X
  
  git add <명시된 폴더 1>
  git add <명시된 폴더 2>
  ...
  
  git status   # ← 명시된 파일만 staged 인지 확인
  git diff --cached --stat   # ← 라인 수 확인
  
  → 다른 모듈 폴더가 보이면 reset 후 다시 add
```

### 4. CLAUDE.md 규칙 21 자동화 안전장치 ✅
```
🔜 11. cowork-staging-lint  →  ✅ 11. cowork-staging-lint 신설 완료 (2026-05-06)
```
