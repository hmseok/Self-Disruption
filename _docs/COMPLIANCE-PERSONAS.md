## 0. 운영 사실 인터뷰 (Rule 25 의무) — 완료 기록

본 모듈은 인터뷰가 이중으로 진행됨 — (1) 사용자 [A]~[E] 답변, (2) 매뉴얼 통합본 5.17 정독. 두 소스 간 불일치는 **매뉴얼이 1차 소스**로 우선 (사용자 동의, 2026-05-18).

### [A] 범위 — 5개 영역 (확정)

개인정보 + 정보자산 + 보안사고 + 규정준수 + 직원교육 — 모두 포함. 통합본 5.17 의 9장 구조에 1:1 매핑 가능.

### [B] 운영 성숙도 — 「체계 성숙기 + 인력 성장기」 (매뉴얼 기반 수정)

- **체계**: 성숙기 — 2019년 제정 → 9회 수정 → 2026.05.20 시행 통합본까지 진화. 9장 27조 + 4개 서식매뉴얼 + 12종 서식 + 연간 운영 캘린더 (RIDE-PLAN-2026) 보유.
- **인력**: 성장기 — CPO 1명 + 관리자 2명 (전담 X, 다른 업무 겸직). 인증 (ISMS/ISMS-P) 보유 여부 미확인 → 사용자 추가 확인 필요.
- **사고 빈도**: 매뉴얼은 4가지 사고 유형 명시 (외부해킹·내부유출·임의변조·법규위반). 실제 발생 빈도는 인터뷰에서 "거의 없음" — 향후 본 모듈로 추적 시작.
- **교육 방식**: 연 2회 이상 (전 임직원), 신규 입사자 별도, 미참석자 보충. 집체 + 인터넷 + 그룹웨어 + 외부위탁 가능 (제23조).

### [C] 자산 등급 — 3단계 (공개 / 내부 / 대외비) (확정 유지)

매뉴얼 통합본 자체가 「문서분류: **내부관리용 (대외비)**」 으로 분류됨 → 3단계 등급 체계와 정합. 자산 등록 폼의 `classification` enum 은 `('public','internal','confidential')` 3 값.

### [D] 역할 — **3-tier** (매뉴얼 기반 수정)

사용자 답변 "4단계 (CISO/보안담당/부서장/직원)" → 매뉴얼 기반 **3-tier** 로 수정 (사용자 동의):

| Tier | 직책 | 매뉴얼 출처 | 라이드 실명 (2026.05) |
|---|---|---|---|
| **1. 개인정보보호 책임자 (CPO)** | 이사급 — 보호체계 총괄, 정기 보고 수령, 자체감사 책임 | 제6조 / 제27조 | **임성민 이사** (라이드케어) |
| **2. 개인정보보호 관리자** | 부장급 — 파기·수시 점검·CPO 보고·취급자 점검 | 제6조 / 제27조 | **석호민 부장 + 양재희 부장** (라이드케어) |
| **3. 개인정보취급자** | 전 임·직원 — 보호조치 이행, 점검 협조, 누설 금지 | 제9조 | 라이드케어 전 임·직원 (정규/임시/계약직 포함) |

**보조 단위**: 「관리팀」 — 침해사고 일선 접수 담당 (제26조 ①). role enum 에 별도 항목으로 보유 가능 (선택 사양 — 사용자 추후 결정).

**부서장 layer 미보유 근거**: 매뉴얼 어디에도 부서장 결재선 명시 없음. 라이드케어가 단일 사업부 단위로 CPO → 관리자 → 취급자 직보고 체계.

### [E] UI — 하이브리드 (확정 유지)

메인 `/RideCompliance` 는 NavTabs 대시보드 — DcStatStrip 5 stat (자산수/사고수/교육이수율/감사현황/연간계획달성률). 상세 deep-link 필요한 페이지는 서브 라우트:
- `/RideCompliance/incidents/[id]` — 침해사고 상세 (대응 절차 추적)
- `/RideCompliance/assets/[id]` — 자산 상세 (이력 + 접근권한)
- `/RideCompliance/audits/[id]` — 감사 결과보고서 상세

리스트·등록 폼은 NavTabs 안에서 처리 (RideVehicleRegistry 동형 — Rule 14).

## 1. 페르소나 1 — 개인정보보호 관리자 (주 페르소나)

### 1.1 프로필
- **실명**: 석호민 부장 (라이드케어 사업부)
- **직무**: 통합본 5.17 제7조 + 제27조 — 파기 실행, 수시 점검, CPO 정기 보고, 취급자 교육 운영
- **도구 (현재)**: 엑셀 + PDF 매뉴얼 + 캐비닛 (서식 종이 보관 3년)
- **도구 (목표)**: `/RideCompliance` 모듈 통합 관리 + 전자결재 연계 (파기·단말기 반출)
- **페인 포인트**:
  1. 14개 도메인 (자산·사고·교육·감사·파기·수탁사·연간계획 등) 이 분산 운영 → 단일 대시보드 부재
  2. 서식 12종이 종이/엑셀 혼재 → 3년 보존 의무 (교육이수확인서/파기이력/감사보고서) 추적 부담
  3. 연간 운영 캘린더 (RIDE-PLAN-2026) 가 PDF 일람표 → 월별 task tracker 필요
  4. CPO 보고 시 KPI 집계 수작업

### 1.2 KPI (대시보드 5 stat)
- 자산 등록 건수 (등급별 분포)
- 침해사고 미종결 건수 (24시간 통지 의무 추적)
- 교육 이수율 (연 2회 × 전 임직원)
- 자체감사 진행 상태 (반기 1회)
- 연간계획 달성률 (월별 task 완료 %)

## 2. 시나리오 — End-to-End (매뉴얼 9장 기반)

### Step 1. 정보자산 등록 (제10조~제18조)
`/RideCompliance/assets` → "자산 등록" 모달
- 자산명 / 유형 (서버·PC·문서·저장매체·CCTV·스마트기기·소프트웨어)
- 등급 (공개·내부·대외비) — Step 2 결정값
- 위치 / 보유자 (assigned_user_id) / 취득일
- 접근 권한 매트릭스 (열람가능 role 다중 선택)
→ `ride_compliance_assets` insert + 변경 이력 log

### Step 2. 침해사고 접수 (제26~27조)
`/RideCompliance/incidents` → "사고 보고" 폼 — **취급자(전 임직원) 누구나 접수 가능**
- 발생일시 / 유형 (4가지 — 외부해킹·내부유출·임의변조·법규위반)
- 영향 범위 (개인정보 항목 + 추정 정보주체 수)
- 초기 조치 내역
→ `ride_compliance_incidents` insert (status='reported') + 관리팀·관리자 알림
- 관리자 단계: 영향 분석 → CPO 보고 → 정보주체 통지 (24시간 의무, 제25조) → status 진행
- 종결: 대응 결과 보고서 작성 → CPO 승인 → status='closed' + 3년 보존

### Step 3. 교육 운영 (제22~23조, 별첨 7 RIDE-PLAN-2026)
`/RideCompliance/trainings` (Phase 1.2)
- 연간 계획 (서식 F-06) → 차수 (1차 2월, 2차 7월) → 대상 (전임직원/취급자/신규/수탁사)
- 이수 확인서 (서식 F-07) 자동 발급 → `ride_compliance_training_records`
- 미참석자 보충 task 자동 생성

### Step 4. 자체감사 + 연간계획 (제20조, 별첨 7)
`/RideCompliance/audits` (Phase 1.2)
- 반기 1회 자체감사 (5월·10월) → 결과보고서 → CPO 보고 → 3년 보관
- 월별 일람표 dashboard — 12개월 task carousel

### Step 5. CPO 정기 보고 (제27조)
`/RideCompliance` 메인 — KPI 5 stat + 미해결 사고/감사 위젯 → CPO 임성민 이사 정기 열람.

## 3. 페르소나 2 — 개인정보보호 책임자 (CPO)

- **실명**: 임성민 이사 (라이드케어)
- **권한**: 모든 데이터 조회 + 자체감사 결과보고서 승인 + 파기 승인 + 정보주체 통지 발송 결재
- **시나리오**: `/RideCompliance` 진입 → 미해결 사고 / 감사 / 파기 승인 대기 위젯 → 결재 / 코멘트
- **UI 차이**: 관리자(석호민) 와 동일 메뉴이나, **승인 액션** 버튼만 노출. 등록·수정은 관리자에게 위임.

## 4. 페르소나 3 — 관리팀 (침해사고 일선)

- **조직 단위**: 라이드케어 관리팀 (제26조 ① 명시)
- **권한**: `/RideCompliance/incidents` 모든 row 조회 + 분류 처리 + 1차 영향 분석 (관리자에게 escalation)
- **시나리오**: 취급자가 사고 보고 → 관리팀에 즉시 알림 → 1차 분류 (4유형) → 영향 분석 → 관리자에게 위임

## 5. 페르소나 4 — 개인정보취급자 (일반 직원)

- **범위**: 전 임·직원 (정규/임시/계약직)
- **권한**: 본인 교육 이수 현황 조회 + 사고 보고 폼 작성 + 본인이 보유한 자산(단말기 등) 조회
- **시나리오**:
  - `/RideCompliance` → "내 교육 이수" 위젯 (본 분기 미이수 표시)
  - 사고 발견 시 "사고 보고" 버튼 (제27조 — "즉시 모든 직원은 관리팀에 사고를 접수")
- **사이드바 노출**: ❌ — 취급자는 별도 진입점 (홈 대시보드 카드)으로 접근. `/RideCompliance` 자체는 관리자 이상 노출.

## 6. 페르소나 5 — 라이드 admin (시스템 관리)

- 모든 권한 + role 매핑 변경 + 매뉴얼 버전 관리
- 본 모듈 외 사이드바·메뉴 등록 영역 — 별도 admin 페이지

## 7. 모듈 외 연계 (V1 — 단순 참조)

- `users.id` (cuid string) — `created_by`, `assigned_user_id`, CPO·관리자 본인 매핑
- `ride_employees` (사이드바 hidden) — 취급자 명단 일괄 동기 (선택 사양, Phase 1.2)
- 기존 FK 관계 **없음** — 라이드 모듈 일관성 (ASSETS / VehicleRegistry 동형)
